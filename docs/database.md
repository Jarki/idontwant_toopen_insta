# Database Tasks

The bot uses SQLite at `data/reels.db` and manages schema changes with Alembic.

Migrations run separately from the bot process. In Docker Compose, a one-shot `migrate` service applies migrations before the long-running `downloader` service starts.

## Default database location

By default, database tasks target:

```text
data/reels.db
```

Do not commit local database files.

## Docker Compose migration service

The default deployment flow runs migrations automatically before the bot:

```bash
docker-compose up -d --build
```

To run only the migration container:

```bash
docker-compose run --rm migrate
```

The `migrate` service uses the same image as the bot and runs:

```bash
/app/.venv/bin/alembic upgrade head
```

It mounts `./data:/app/data`, so it targets the same SQLite database as the bot.

## Poe commands

For local development, run database tasks through `uv run poe`:

```bash
uv run poe db-upgrade              # apply migrations up to head
uv run poe db-current              # show the current revision
uv run poe db-history              # list migration history
uv run poe db-downgrade            # downgrade one revision
uv run poe db-revision "message"   # create an autogenerated revision
```

## Targeting another SQLite database

For manual commands, set `DB_PATH` or `DATABASE_URL`.

Use `DB_PATH` for a local SQLite file:

```bash
DB_PATH=/tmp/reels.db uv run poe db-upgrade
```

Use `DATABASE_URL` when you need an explicit SQLAlchemy URL:

```bash
DATABASE_URL=sqlite:////tmp/reels.db uv run poe db-current
```

`DATABASE_URL` takes precedence when both variables are set.

## Creating a migration

When a schema change is needed:

1. Update the SQLAlchemy model/table definition in `ig_reel_downloader/repository/sqlite.py`.
2. Create a migration:

   ```bash
   uv run poe db-revision "describe schema change"
   ```

3. Review the generated file in `migrations/versions/` before committing it.
4. Run the migration against a disposable database:

   ```bash
   DB_PATH=/tmp/reels.db uv run poe db-upgrade
   ```

5. Run the full validation gate:

   ```bash
   uv run poe verify
   ```

## Existing database behavior

The initial migration is designed to preserve existing data:

- If an unversioned database already has the expected `reels` table, Alembic baselines it without deleting rows.
- If the existing `reels` table is missing required columns, migration fails instead of stamping an incompatible schema.
- The initial downgrade preserves `reels` data.

## Operational notes

- The bot process does not apply migrations at startup.
- Docker Compose applies migrations with the one-shot `migrate` service before starting `downloader`.
- SQLite remains the only storage system unless the user explicitly approves a different storage architecture.
- Cleanup removes old media files from `output/`; it does not prune SQLite rows.
- Cached rows are reused only while fresh and while the referenced media file still exists.
